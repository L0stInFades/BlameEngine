# RHI Thin Layer And Metal Backend Split

This records the first RHI boundary and the Metal backend split that preserves the current `Renderer` API while creating room for renderer experiments.

## Public Boundary

`engine/rhi` is intentionally thin. It defines stable renderer vocabulary, not a full render graph:

- `next/rhi/types.h`: backend, queue class, color/depth/depth-stencil format, vertex format/step function, index format, extent, clear color, resource memory/type.
- `next/rhi/device.h`: backend device identity, feature flags, supported queue classes, and dedicated queue classes.
- `next/rhi/swapchain.h`: drawable size and color/depth formats.
- `next/rhi/command.h`: command context state and submitted frame index, plus small indexed draw, index-buffer-view, viewport, scissor, draw-time stencil-reference, and blend-constant descriptors used by backend draw paths.
- `next/rhi/resource.h`: small resource descriptors and base resource metadata.
- `next/rhi/pipeline.h`: small graphics pipeline descriptors, validation, and descriptor-key equality for shader entry points plus vertex input layout, color/depth/stencil/raster state, fill mode, depth-bias/depth-clip state, multisample state, and per-target color blend/write-mask state.
- `next/rhi/render_pass.h`: small render pass color/depth/stencil attachment descriptors, load/store actions, and validation for attachment count, format capability, clear depth/stencil, and action enums; Metal now has a tiny builder that turns this descriptor plus native textures, including optional native color/depth resolve targets, into an `MTLRenderPassDescriptor` before a full frame graph lands.
- `next/rhi/frame_graph.h`: small frame-graph resource/pass/access descriptors with explicit ordered pass dependencies, optional shader-stage hints, fixed-capacity pass dependency/resource tables, queue-aware resource-state validation, duplicate-dependency and duplicate-access rejection, shader-stage-mask validation plus stage/queue compatibility checks, a backend-neutral transition planner that records pass/access/resource indices plus before/after states, compiled pass dependency/access tables plus per-pass dependency/access/transition ranges with completeness helpers for invalid-entry and non-contiguous-range rejection, Metal command scope diagnostics and native render-encoder resource-use declarations that consume those ranges, and a live-resource transition bridge that rejects incomplete compiled tables before sequentially applying whole-frame or single-pass compiled plans to bound `RHI::Resource` objects.
- `next/rhi/shader_resource_group.h`: small ShaderResourceGroup layout descriptors with typed constant-buffer, buffer, texture, storage-texture, and sampler binding ranges, shader-stage flags, range helpers, and descriptor validation for invalid ranges and stage-aware binding overlap before a real SRG/argument-buffer backend lands.
- `next/rhi/swapchain.h`: swapchain descriptor validation for drawable size plus color/depth attachment formats before backend layer setup.

The goal is to let DX12, Metal, and future experimental backends converge on common names before NEXT commits to a full frame graph or material system.

## Metal Split

The public `Next::Renderer` contract is unchanged. Internally the macOS backend now uses focused pieces under `engine/renderer/src/metal`:

- `MetalDevice`: owns `id<MTLDevice>`, the graphics/upload queue objects, queue-class lookup, dedicated queue reporting through `RHI::DeviceFeatures`, device name, and feature flags such as unified memory and argument buffer tier, with Apple-only tests covering queue feature flags, native queue lookup, argument-buffer tier mapping, argument-buffer helper consistency, and shutdown cleanup.
- `MetalSwapchain`: owns `CAMetalLayer`, drawable acquisition, render pass descriptor, drawable size, descriptor-backed depth target resize, and swapchain counters for resize/acquire/present/release diagnostics, with RHI descriptor validation and Apple-only tests covering invalid descriptor rejection, invalid acquire stats, RHI render-pass descriptor building, explicit present-pass dependency on the render pass, plus layer/depth resize lifecycle.
- `MetalRenderPipeline`: owns descriptor-backed `MTLRenderPipelineState` and `MTLDepthStencilState` creation/binding from `RHI::GraphicsPipelineDesc`, labels both native states from the RHI debug name for GPU capture diagnostics, exposes internal native-state accessors for targeted Metal tests, and maps `MTLVertexDescriptor` vertex input layout, triangle fill mode, depth-bias/slope/clamp and depth-clip encoder state, multisample sample count, alpha-to-coverage, per-target color blend factors/operations/write masks, and optional front/back stencil state for merged depth-stencil formats; `MetalPipelineCache` reuses equivalent descriptors through RHI-owned hash/equality keys and reports request/hit/miss/failure counts through renderer pipeline snapshots.
- `MetalCommandContext`: owns the command buffer lifecycle, begins from `MetalDevice` queue-class lookup, records queue class identity, gates render pass encoding and present to graphics contexts, supports non-present command buffer commits, rejects incomplete frame-graph compile tables and native resource-use tables before encoding dependency/access/transition scopes, rejects native resource-use tables whose resource count does not match the compiled graph, rejects native resource objects whose Metal class or table type does not match the compiled RHI resource type, and tracks begin/render-pass/commit/present counters plus submitted frame index, with Apple-only tests covering success and rejection counters; the render path now builds RHI viewport/scissor descriptors from the swapchain drawable size, exposes that draw-state snapshot through public renderer diagnostics, and maps the same descriptors to `MTLViewport` / `MTLScissorRect` before draw submission.
- `MetalBufferPool` / `MetalTexturePool`: provide the current resource creation and release entry points, tracking live/peak resource counts and bytes plus budget rejection counters, with Apple-only tests covering allocation, release, deferred native teardown, and per-memory budget failures before a real allocator lands.
- `MetalBuffer` / `MetalTexture`: own descriptor-backed native Metal resources, including RHI sample-count-backed 2D multisample render targets, and defer teardown through `MetalReleaseQueue` when a submitted frame index is available, with Apple-only tests covering descriptor retention, debug labels, native handles, state mutation, invalid-descriptor preservation, MSAA native texture shape, and deferred native release.
- `MetalUploadQueue`: owns reusable staging frame packets backed by `MetalBufferPool`, a mutex-guarded pending buffer/texture/uniform request queue, automatic packet-full batch flushing, batched blit submission, serial upload handles, recent per-serial completion status history, and JobSystem-backed async wait jobs with completion callbacks, with Apple-only tests covering buffer/texture copies, invalid upload rejection, status history, callbacks, and packet-full auto-flush.
- `MetalUploadQueue` reports whether its submissions are copy-queue or graphics-queue backed, and Metal resource state tracking now uses that queue class for upload-side `CopyDestination` transitions before returning resources to graphics-visible states.
- Upload handle completion now resolves through exact packet/history status instead of treating the highest observed serial as proof that every lower serial is complete; failed uploads are considered finished for promotion cleanup but not successful waits.
- Upload completion counting now tracks successful completions exactly and de-duplicates repeated Metal completion-handler / wait-path observations.
- Upload frame-packet reuse waits for Metal command-buffer status instead of `waitUntilCompleted`, while explicit and async upload waits now sleep on completion-handler status notifications after snapshotting the live command buffer, avoiding completion-handler re-entry deadlocks during batched texture uploads.
- Upload handles that are no longer in a live frame packet or retained status history report `Unknown` instead of being inferred as permanently pending.
- Async upload waits now invoke callbacks immediately for retained completed/failed handles and otherwise sleep on completion-handler status notifications for live uploads.
- RHI now exposes `SamplerDesc` plus filter/address/mip/compare/border-color enums, explicit anisotropy enable/max fields, and LOD clamp/bias fields; Metal material sampler creation goes through `metal_conversions.*` instead of local `MTLSamplerDescriptor` literals, with Apple-only conversion tests covering sampler descriptor fields and anisotropy fallback.
- Metal material draws now bind a fixed sampler table matching the five material texture slots, replacing the previous single sampler state rebound at every slot.
- Metal material sampler states are now cached by `MetalSamplerCache` using `RHI::SamplerDescEquals`, which ignores debug labels, so identical fixed-slot sampler descriptors share one `MTLSamplerState` while retaining distinct shader binding slots; `RendererSamplerStats` exposes fixed material sampler table readiness, bound-slot count, and cached sampler count for smoke/tooling diagnostics; Apple-only tests cover descriptor reuse, distinct descriptor allocation, deferred sampler release, and scene-resource sampler snapshots.
- `MetalShaderLibrary`: loads the demo forward shader through a descriptor-driven library boundary. `MetalShaderLibraryManifest` parses `demo_forward.shader_manifest` into version/debug/metallib/source paths with at least one non-empty normalized manifest-relative source or metallib path plus required `vertexEntry` / `fragmentEntry` names and optional `materialLayout` / `pipelineLayout` / `requiredArgumentBufferTier` metadata, rejects absolute or escaping manifest input paths, empty declared input or entry values, invalid argument-buffer tier names, duplicate known keys, and malformed or unsupported manifest versions beyond v1. The demo forward path requires `materialLayout=material_srg_v1`, `pipelineLayout=demo_forward_pipeline_v1`, and `requiredArgumentBufferTier=tier1`, so runtime-dir or explicit demo manifests that do not declare the current material SRG ABI, demo pipeline-layout ABI, and hardware capability contract fail fast instead of falling back to stale shader packages. `MetalShaderLibraryDesc` carries the manifest version/path/debug name plus shader inputs, entry points, material layout, pipeline layout, and required argument-buffer tier to native library and RHI pipeline creation, `SelectShaderLibraryInput` prefers an existing `.metallib` and otherwise requires an existing source file, `CreateShaderLibrary` resolves both requested `MTLFunction` entry points before marking the library input ready, labels the resulting `MTLLibrary` with the manifest debug name for GPU capture diagnostics, and the selected input owns its path for stable renderer diagnostics. `MetalSceneResources` retains the active shader desc so RHI pipeline entry-point pointers stay valid, checks the manifest-required argument-buffer tier against `MetalDevice` features before shader/pipeline creation, and renderer pipeline snapshots can expose the manifest version/path/debug name/material layout/pipeline layout/required tier plus manifest-driven entry names. CMake stages the demo `.metal` source and manifest into the build-tree shader directory and points the compile-time `NEXT_METAL_SHADER_DIR` fallback there, so default runtime loading uses package-shaped staged assets instead of the source tree. At runtime, `NEXT_METAL_SHADER_DIR` can point at a packaged shader directory; otherwise the loader searches shader directories beside the executable and its parents before using the compiled staged fallback. Any discovered manifest file is authoritative and must parse successfully and satisfy the demo material/pipeline-layout/tier contract; invalid runtime-dir or executable-adjacent manifests no longer fall back to default shader inputs. Full Xcode toolchains get an optional CMake-built `demo_forward.metallib` loaded with `newLibraryWithURL`; Command Line Tools-only setups fall back to compiling the staged `demo_forward.metal` with `newLibraryWithSource`. Packaged and CI runs can override the demo manifest with `NEXT_METAL_DEMO_FORWARD_MANIFEST`, or override the demo inputs with `NEXT_METAL_DEMO_FORWARD_METALLIB` and `NEXT_METAL_DEMO_FORWARD_SOURCE`; input override paths are normalized before descriptor/stat reporting, empty overrides are ignored, and a non-empty explicit manifest override fails fast when missing, invalid, or missing the current material layout, pipeline layout, or required argument-buffer tier rather than silently falling back. A stale, invalid, or entry-point-mismatched `.metallib` logs a warning and falls back to source when that source exists; a source library whose compiled functions do not match the manifest entries fails before pipeline creation.
- The demo shader manifest also declares `materialShaderResourceGroupArgumentBufferIndex=2` plus the material SRG argument-buffer resource IDs `materialShaderResourceGroupUniformArgumentIndex=0`, `materialShaderResourceGroupTextureArgumentBaseIndex=1`, and `materialShaderResourceGroupSamplerArgumentBaseIndex=6`. `MetalShaderLibraryDesc` carries that fragment argument-buffer slot and internal resource layout alongside the material/pipeline layout and hardware tier metadata, `MetalSceneResources` checks them against the renderer's public material SRG slot and generated argument descriptors before creating draw resources, and pipeline stats plus `song_demo` shutdown logs expose the selected `2` and `0/1/6` values so packaged shader metadata and render encoder binding stay auditable.
- `MetalSceneResources`: owns the current demo pipeline, depth state, buffers, material textures, neutral fixed-slot fallbacks, sampler, and per-frame uniform storage, and now requests the shader library through `MetalShaderLibrary` instead of carrying an inline MSL string.
- `MetalReleaseQueue`: defers Objective-C Metal object releases by the device frame latency, following O3DE Atom's collector shape at NEXT scale, and reports queued/collected/pending/forced-collect counters for lifetime diagnostics.

`metal_renderer.mm` is now orchestration: it converts `RendererFrameDesc` into uniforms, asks the split pieces to acquire, encode, draw, present, and resize.

## Deliberately Deferred

- No full frame graph executor yet. The current RHI boundary now has pass dependency/resource access descriptors, transition planning, compiled pass dependency/access tables, per-pass dependency/access/transition ranges, a Metal command-scope consumer for render/present pass ranges, Metal render-encoder resource-use declarations for compiled non-attachment accesses, and a live-resource transition bridge, including Metal demo resource readiness plus swapchain render/present planning with scene draw resources attached, but Metal and DX12 still execute the demo through their existing direct paths until pass compilation owns command submission and backend barrier encoding.
- No full shader asset pipeline yet. The demo now has a sidecar manifest, staged `.metal` source, optional CMake-built `.metallib`, and runtime entry-point validation, but there is still no general shader asset registry, variant compilation, hot reload, or package-wide shader dependency tracking.
- No asset-streaming upload scheduler yet. `MetalUploadQueue` now has a thread-safe frame-packet ring, pending request batching, packet-full auto-flush, completion handles, recent status history, and async wait callbacks. `AssetManager` can now pump completed async loads on the main thread, reports pending load jobs plus completed callback backlog through `AssetStats`, and feeds observable nonblocking material texture uploads with renderer-level handle/status queries and source-asset identity. There is now a fixed-capacity renderer material binding table, per-slot texture registry, a fixed material SRG argument-buffer path for all five texture slots, typed sampler descriptors, a descriptor-keyed Metal sampler cache, and a fixed Metal sampler binding table, but not yet a dynamic sampler array, bindless material table, or general shader resource group layer.
- No direct O3DE class port. The split mirrors useful Atom responsibilities without importing AzCore/AZStd contracts; the release queue is a compact local analogue of Atom's `ObjectCollector`-backed queue.
