# O3DE Code Intake And Metal Adaptation

This note records the first useful O3DE code intake for NEXT and the practical path for adapting Atom's Metal ideas without importing the whole Atom stack.

## Current NEXT Metal Shape

`engine/renderer/src/metal/metal_renderer.mm` is intentionally still a compact backend:

- one `id<MTLDevice>`
- one `id<MTLCommandQueue>`
- one `CAMetalLayer`
- demo forward Metal shader source loaded from `engine/renderer/shaders/metal/demo_forward.metal` and compiled with `newLibraryWithSource`
- local buffers, texture, depth target, and render pipeline creation
- descriptor-backed Metal buffer/texture wrappers and thin pool entry points for demo mesh, uniforms, material texture, and swapchain depth resources
- a reusable pooled frame-packet staging/blit resolver that batches the material texture, demo mesh buffer uploads, and per-frame uniform/debug-cell updates into private Metal resources
- frame-latency deferred release of old Metal objects during resize and renderer teardown
- direct frame flow through drawable, command buffer, render encoder, present, with renderer-visible command-context and swapchain counters for smoke diagnostics

That is enough for the current `song_demo`, but it is not enough for renderer-scale features like GPU upload scheduling, frame graph passes, async streaming uploads, bindless resources, material pipelines, or multiple hardware queue classes.

## Recommended Order

1. Add a thin `engine/rhi` module with public interfaces for device, swapchain, command list, buffer/image, pipeline, and render graph handles.
2. Split the current Metal backend into `MetalDevice`, `MetalSwapchain`, `MetalCommandContext`, and small resource helpers while keeping the existing cube output unchanged.
3. Keep moving the MSL path toward a packaged shader asset flow; CMake can now build an optional demo `.metallib`, runtime fallback still compiles source when offline tools are unavailable, and public pipeline stats expose manifest/path/entry/cache helper flags for smoke tooling.
4. Continue broadening shader package metadata beyond the demo manifest now that the material SRG and demo pipeline-layout ABIs are explicit and stale packages fail before native shader-library selection.
5. Connect the new RHI frame graph pass/access descriptors to backend barrier encoding and command submission after resource lifetime and command submission stay stable.
6. Bring in advanced O3DE-inspired rendering features only behind this RHI layer.

## Risks

- Atom is deeply tied to AzCore/AZStd and O3DE's RHI contracts. Port architecture, not direct class bodies.
- Metal argument buffers and bindless behavior vary by feature tier; NEXT now records the material SRG Tier1 requirement in the demo shader manifest and checks it against device features before pipeline creation, while future bindless paths should stay behind explicit Tier2 checks.
- Shader pipeline differences are a real project item: O3DE's shader build flow should be studied, and NEXT still needs a general shader asset pipeline beyond the current demo manifest, staged `.metal`, and optional `.metallib` path.
- Resource lifetime is the first correctness risk once streaming uploads meet rendering. A release queue/fence model should land before heavy renderer features.
